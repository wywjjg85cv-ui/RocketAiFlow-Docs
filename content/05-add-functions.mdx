# Add Functions

Functions give the agent controlled ways to act during the call.

For a first implementation, keep the function set small and directly tied to the workflow you are
testing.

## Recommended First Functions

Start with:

- `hangup_call`
  Use this when the call is complete and should end cleanly.

- `transfer_call`
  Use this when the caller should move to a human extension or destination.

- optional: `get_weather`
  Use a simple custom API function only if you want to test function-calling behavior without
  adding business-critical dependencies.

## Transfer Setup Note

For transfer flows, use `exten` as the main parameter when you need to pass the destination
extension.

This keeps the handoff logic explicit and easy to review during tests.

## How To Decide What Belongs In A Function

Use a function when the agent needs to:

- transfer the call
- end the call
- request structured information from another system

Functions are the practical bridge between the conversation and the rest of the business workflow.
This is where RocketAiFlow becomes more than a static voice script.

Do not add functions for behavior that can be handled by prompt instructions alone.

## Functions can trigger API-backed actions

Functions can support external actions such as:

- updating a CRM after lead qualification
- checking availability before appointment booking
- looking up account or contact details
- sending a follow-up action after a call outcome is confirmed
- routing the call based on an external rule or decision service

The goal is not to add APIs everywhere. The goal is to add functions only where the workflow needs
structured interaction with another system.

## Dynamic values belong in function inputs

Function inputs do not have to stay static. They can receive context-specific values such as:

- contact fields
- workflow variables
- campaign identifiers
- route or call metadata
- runtime values collected during the conversation

This is useful when one function should behave differently depending on who is calling, which
campaign is running, or what the agent has already learned during the call.

## Function design best practices

- keep the function scope tied to one business action
- pass only the parameters required for that action
- use names that make sense to operators and implementers
- keep descriptions specific enough that prompt writers know when to use the function
- avoid combining unrelated actions into one endpoint or one function schema

## First Validation Pass

After saving the functions, confirm:

- the function names are clear
- required parameters are present
- transfer destinations are valid
- the prompt tells the agent when to call each function

## Common Function Mistakes

- adding too many functions before the first test
- using unclear parameter names
- forgetting to align the prompt with the available functions
- testing transfers without a valid destination
