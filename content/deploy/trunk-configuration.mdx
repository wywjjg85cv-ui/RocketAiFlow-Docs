# Trunk Configuration

Trunk configuration supports the calling path used by both inbound routing and outbound campaigns.

If the trunk is not aligned with the rest of the setup, routing and campaign behavior can look
incorrect even when the agent itself is configured properly.

RocketAiFlow is designed for provider flexibility, but provider choice depends on deployment and
configuration. Trunk setup is where that flexibility becomes practical rather than abstract.

## When To Review Trunk Setup

Review trunk configuration before:

- connecting a live inbound route
- launching an outbound campaign
- diagnosing calls that never start or never reach the intended destination

## What The Trunk Needs To Support

At a minimum, confirm that the trunk setup matches the workflow you are testing:

- inbound calling if the agent should answer inbound routes
- outbound calling if the agent should run campaigns
- the expected destination or source context for the first test
- the SIP / PBX environment you are integrating with

## Why the telephony layer matters

RocketAiFlow uses an Asterisk-based telephony layer. The telephony architecture is built on
Asterisk-based foundations to support practical SIP / PBX integration, trunk-based connectivity,
and provider flexibility.

In practice, that means:

- RocketAiFlow is built for real SIP and PBX environments
- telephony interoperability should be validated during setup
- the right provider fit depends on deployment and configuration
- inbound and outbound workflows both rely on the same underlying telephony path

## Recommended First Validation

Before scaling up, check:

- the trunk is active and reachable
- inbound and outbound paths match the intended use case
- the route or campaign depends on the correct trunk context

## Common Trunk-Related Symptoms

- inbound calls never reach the agent
- outbound calls do not start
- transfers fail even though the agent asked for them
- call records show repeated failures without clear prompt issues
- the selected provider or PBX setup needs deployment-specific validation

## Operational visibility for trunk issues

RocketAiFlow can include a ready Grafana monitoring foundation so teams do not have to assemble the
first telephony-related dashboards from scratch.

For trunk and telephony investigations, the most useful starting points are usually:

- `Asterisk Overview` for higher-level telephony behavior
- `Asterisk Log` for telephony-related log activity
- `Node Exporter Full` when host conditions may be affecting call handling

This helps teams confirm whether the problem belongs to the telephony path before they change
agent, routing, or campaign configuration.
