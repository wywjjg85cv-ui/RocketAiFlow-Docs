# Monitoring and Visibility

RocketAiFlow is not just a workflow builder. It is an operational product. Teams running inbound
and outbound voice workflows at scale need real-time visibility into what is happening now, what
changed recently, and where performance is drifting.

That includes the telephony path. RocketAiFlow uses an Asterisk-based telephony layer, so
observability needs to cover not only workflow behavior but also the practical SIP / PBX
environment behind inbound and outbound calling.

## Why monitoring matters

Monitoring helps teams:

- detect issues faster before they turn into missed outcomes
- understand campaign and endpoint behavior while calls are active
- compare live load with configured limits
- see whether outcomes are shifting over time
- make better optimization decisions with operational evidence

## What teams usually monitor

The exact implementation depends on whether RocketAiFlow runs in a Dedicated or Self-Hosted model
and on the monitoring stack, but practical coverage often includes:

- answered calls
- no answer activity
- busy calls
- failed calls
- congestion events
- total call volume
- active calls versus configured call limits
- campaign active state
- dialing status and pacing
- endpoint and trunk state
- average talk time
- average ring time
- performance trends across selected time windows

## Product-led monitoring view

Keep the dashboard language useful for operators and managers. On the product side, present
monitoring as business-readable categories such as:

- call outcomes in real time
- live campaign monitoring
- endpoint and trunk health
- active load and pacing
- performance insights over time

The raw metric names can stay in the technical layer.

## Pre-provisioned Grafana dashboards

RocketAiFlow can come with a ready Grafana dashboard foundation, so teams do not need to build the
core monitoring layer from scratch before they can see useful operational signals.

That pre-provisioned starting point can include dashboard categories such as:

- `Node Exporter Full`
- `Asterisk Log`
- `Asterisk Overview`
- `cAdvisor`
- `MySQL 8`

Together, these give teams faster visibility into:

- host and system health
- telephony activity
- Asterisk behavior and logs
- container runtime monitoring
- database visibility

This is especially important for a flexible telephony architecture, where provider choice depends
on deployment and configuration and should be validated with operational evidence.

<Callout type="info">
  Treat the pre-provisioned Grafana set as the first operational baseline. Teams can extend it
  later, but they do not need to invent the initial observability layer just to begin operating the
  platform responsibly.
</Callout>
