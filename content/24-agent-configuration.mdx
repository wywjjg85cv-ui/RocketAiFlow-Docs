# Agent Configuration

RocketAiFlow agents are designed to be quick to configure and practical to extend. The goal is not
to force every workflow into one rigid template. The goal is to help teams configure agents for real
business cases and then add only the capabilities the workflow actually needs.

## What agent configuration includes

In practical terms, agent configuration usually means:

- defining the agent role and business objective
- setting the language and conversational scope
- configuring the prompt and operational behavior
- deciding which functions the agent can use
- preparing the agent for inbound routes, outbound campaigns, or both

## A simple configuration path

Use this order when you are setting up a new agent:

1. define the business use case
2. create the agent with a clear name and role
3. write the prompt around that single use case
4. attach the functions required for the action path
5. test the agent inside one route or one campaign

## Start from the business case

The most useful first question is not “what settings are available?” It is “what job should this
agent actually do?”

Typical examples:

- lead qualification for inbound or outbound sales follow-up
- appointment booking and scheduling intake
- routing decisions based on caller intent
- CRM updates after a structured information capture
- external lookup actions during a call

## Keep the first agent focused

A focused agent is easier to prompt, easier to test, and easier to troubleshoot.

Good examples of narrow first-agent scope:

- qualify demo requests and transfer qualified leads
- collect booking details and trigger a scheduling action
- answer first-line questions and route to the correct team

Avoid building one agent that tries to behave like support, sales, billing, and dispatch at the
same time.

## How configuration connects to functions

Agent configuration should make the function layer easier to reason about.

That means:

- the agent role should explain why a function exists
- the prompt should explain when the function should be used
- the function inputs should reflect the information the workflow can actually provide
- the agent should only have access to actions that fit the business case

For example, a lead qualification agent may need `transfer_call` and `update_crm_lead`, while an
appointment assistant may need `check_availability` and `book_appointment`.

## Real workflow examples

### Lead qualification

- collect name, company, and interest level
- decide whether the lead qualifies for a human follow-up
- update a CRM or route the caller to sales

### Appointment booking

- collect service type, preferred time, and contact details
- call a booking function or availability API
- confirm the next step or escalate when booking is not possible

### Routing assistant

- identify why the caller is calling
- use a small function set for transfer or lookup
- route the call to the correct team with clear handoff logic

## Configuration best practices

- keep the agent goal clear and specific
- design one agent around one primary workflow
- attach only the functions needed for that workflow
- keep naming consistent across agent, prompt, and function descriptions
- expand the configuration only after a working test path exists
