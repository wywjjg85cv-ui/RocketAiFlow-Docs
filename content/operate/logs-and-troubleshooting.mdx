# Logs and Troubleshooting

Troubleshooting live voice automation requires both conversation context and operational context.
The question is not only what happened on one call. It is also what the broader system was doing at
the time.

## Evidence to collect first

Start with:

- call and workflow identifiers
- campaign status at the time of the issue
- answered, no answer, busy, failed, and congestion outcomes around the incident window
- active calls versus configured limits
- endpoint and trunk state
- relevant timestamps for talk time, ring time, transfer events, or API failures

## A practical investigation flow

1. Confirm the workflow path the call took.
2. Review campaign activity and call outcome shifts around the same time.
3. Check endpoint and trunk state for matching telephony issues.
4. Confirm whether the issue belongs to workflow logic, data quality, provider behavior, or runtime state.
5. Apply one change at a time and re-test with a controlled scope.

## Common patterns

### Outcome mix changed suddenly

Check whether no answer, busy, failed, or congestion outcomes shifted before changing workflow
logic. Many apparent product issues start with changes in telephony or contact conditions.

### Active calls are low but campaigns look stalled

Check campaign status, configured limits, and dialing pace together. A low active call count does
not necessarily mean the workflow is healthy.

### Endpoint or trunk instability

Review endpoint state timeline and trunk readiness before changing prompts or routing logic. The
telephony path may be the actual cause.

## Use the pre-provisioned dashboards first

RocketAiFlow can include pre-provisioned Grafana dashboards that give teams a faster starting point
for investigation:

- `Node Exporter Full` for host and system health
- `Asterisk Overview` for higher-level telephony monitoring
- `Asterisk Log` for telephony-related log review
- `cAdvisor` for container runtime monitoring
- `MySQL 8` for database operational checks

This means teams can move from symptom to evidence without first building the basic dashboards
themselves.

## Monitoring stack note

If the deployment uses Prometheus and Grafana, use those tools to correlate operational metrics and
time windows during investigation. Keep the documented behavior aligned with the monitoring stack
that is actually configured.
