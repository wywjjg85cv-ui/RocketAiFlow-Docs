# Platform Overview

RocketAiFlow is organized around the practical steps required to launch and review AI voice
workflows for inbound and outbound calls.

The product is easiest to understand as a connected flow:

1. Create an AI voice agent
2. Define how that agent should speak and act
3. Add the functions required for transfer, hangup, or external actions
4. Connect the agent to inbound routing or outbound campaigns
5. Import contacts when outbound calling is needed
6. Test the call behavior
7. Review the result in call records

## Practical agent extension model

RocketAiFlow is designed so teams can configure agents quickly and then extend them without turning
the workflow into a rigid one-off setup.

In practical terms, teams can:

- configure an agent around one real business objective
- define prompt behavior and handoff rules
- attach only the functions that support the workflow
- call external APIs when the workflow needs an outside action
- pass dynamic values into function inputs based on contact, call, route, or campaign context

That makes the platform easier to adapt across business cases such as lead qualification,
appointment booking, routing decisions, CRM updates, follow-up actions, and external lookups.

## Main Product Areas

### AI Voice Agents

Use this area to define the agent itself. This is where you set the name, language, behavioral
goal, prompt, and operational tools that support the conversation.

Agent configuration does not stop at the prompt. In practical workflows, the agent definition,
function access, and runtime context all work together.

### AI Inbound Routing

Use this area to decide which agent should answer a given inbound route. This is the bridge
between your telephony entry point and the AI conversation layer.

### AI Dialer Flows

Use this area to create outbound campaigns. A campaign typically brings together an agent, a target
contact set, and the rules for when outbound calls should run.

### Import Contacts

Use this area when an outbound flow needs a prepared list of contacts. A clean import step reduces
campaign errors and makes testing more predictable.

### Call Records

Use this area to validate behavior after tests and live calls. This is where you confirm whether a
call reached the intended outcome.

### Operational Visibility

Use the monitoring layer to understand what is happening across live voice workflows after launch.
This includes answered and unanswered outcomes, active calls versus configured limits, campaign
activity, dialing pace, endpoint state, and performance trends over time.

RocketAiFlow can also include pre-provisioned Grafana dashboards, giving teams a ready
observability starting point instead of requiring them to assemble the first dashboard layer from
scratch.

### Trunk

Use this area to connect the calling environment that supports inbound and outbound activity. Trunk
configuration needs to line up with routing and campaign behavior.

### Phone

Use this area for manual testing from inside the platform. It is useful when you want to validate
agent behavior before expanding to wider routing or campaign use.

## How The Areas Work Together

- `AI Voice Agents` defines who is speaking.
- `Prompt` and `Functions` define how the agent behaves.
- `AI Inbound Routing` decides where inbound calls go.
- `AI Dialer Flows` decides how outbound calls are organized.
- `Import Contacts` provides the targets for outbound work.
- `Phone` helps you test the setup.
- `Call Records` tells you what actually happened.
- `Trunk` supports the telephony layer behind both inbound and outbound usage.

## A Practical Setup Pattern

For a first implementation, use a narrow scope:

- start with one agent
- add only the prompt and functions needed for the first use case
- connect one inbound route or one small outbound campaign
- test with a limited number of calls
- review the result in call records before expanding

This reduces ambiguity and makes troubleshooting much faster.

## Operational Review Questions

Before moving past setup, confirm:

- which agent should handle the call
- what the agent is allowed to do
- when a human transfer should happen
- how inbound calls reach the agent
- how outbound campaigns receive contacts
- where call outcomes will be reviewed

## Monitoring And Visibility

Operational review does not stop at call records. Teams running inbound and outbound workflows at
scale need live visibility across the current operating state.

Monitor areas such as:

- answered calls, no answer activity, busy calls, failed calls, and congestion events
- total calls and active calls compared with configured call limits
- campaign active state and dialing status
- endpoint and trunk state
- average talk time and average ring time
- time-range performance trends and outcome distribution across campaigns
