# Functions and API Calls

RocketAiFlow functions let agents do more than speak. They let the workflow trigger structured
actions that connect the conversation to the rest of the business process.

That can include built-in call actions such as transfer and hangup, or custom actions backed by
external APIs.

## What functions are used for

Use functions when the agent needs to:

- perform a call control action
- request or update data in another system
- trigger a business action outside the conversation
- send structured values gathered during the call

## Practical use cases

Common function-based workflow actions include:

- lead qualification updates in a CRM
- appointment availability checks
- routing decisions based on external rules
- follow-up task creation after a call
- external lookups for account, location, or service information

## A practical setup path

1. identify the real business action
2. define the smallest useful function for that action
3. list the parameters the action actually needs
4. tell the prompt when the function should be used
5. test the function with one controlled workflow

## How API-backed functions fit the workflow

An API-backed function usually follows a simple path:

1. the agent collects the needed information during the call
2. the agent calls a function with structured inputs
3. the function triggers an external API endpoint
4. the API returns a result the workflow can use
5. the agent continues the conversation based on that result

## Example: CRM update after lead qualification

The agent may collect:

- caller name
- company
- interest level
- preferred follow-up

Then the function sends those values to an external CRM endpoint so the lead is recorded without
manual re-entry.

## Example: appointment booking

The agent may collect:

- service type
- requested date
- requested time range
- contact phone

Then the function can call an availability or booking API and return the next step to the caller.

## Dynamic values matter

Function inputs can be built from:

- dynamic fields
- workflow variables
- contact fields
- runtime values collected during the call
- contextual parameters such as route or campaign identifiers

This is how one function can stay reusable while still adapting to different business cases.

## Function design best practices

- design functions around real business actions, not generic technical verbs
- pass only the parameters required for the action
- keep function names and descriptions easy to understand
- align prompt instructions with the available function set
- keep API actions small enough to test and maintain

## Common mistakes

- exposing too many functions to one agent
- mixing unrelated actions in one function
- sending more parameters than the action needs
- hiding business meaning behind unclear function names
- calling external APIs before the required information has been collected
