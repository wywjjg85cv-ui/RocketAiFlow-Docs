# Dynamic Parameters

Dynamic parameters let RocketAiFlow functions receive values that depend on the current workflow,
contact, call, or business case.

This makes function calls more reusable and more realistic than hardcoded inputs.

## What dynamic parameters mean

Instead of sending the same fixed input every time, a function can receive values such as:

- contact fields
- workflow variables
- route or campaign context
- runtime values collected during the conversation
- call-specific identifiers or metadata

## Why they matter

Dynamic parameters help the same function support multiple real workflows without rewriting the
logic for every case.

Examples:

- a lead qualification function can receive the current contact phone and the qualification result
- a booking function can receive the date collected during the call plus the campaign or route context
- a CRM update can receive the caller details and the runtime outcome of the conversation

## Practical parameter sources

## Contact template variables [#contact-template-variables]

Contact templates make agent text and workflow context dynamic. Instead of writing fixed values in
the greeting, prompt, or function context, you reference variables that are replaced with data from
the current contact.

The default template includes the contact parameters that belong to a contact created in
RocketAiFlow:

- `{t.CampaignId}`
- `{t.ContactId}`
- `{t.id}`
- `{t.name}`
- `{t.phone}`
- `{t.priority}`
- `{t.surname}`
- `{t.totalGlobal}`

You can edit the default template and add more keys. Custom keys are rendered from `data`.
For example, if you add a key named `cf` in the template, the variable is rendered as
`{t.data.cf}`.

This prevents typing mistakes when the customer chooses variables to render dynamically, because
the UI can suggest the variables defined by the selected template.

The `data` field is the flexible container for custom customer fields. Use it when the outbound
contact list includes business-specific key-value pairs such as city, plan, CRM notes, lead type,
product interest, preferred branch, fiscal code, or any other context the agent should know during
the call.

In agent fields that support template variables, type `@` to open variable suggestions, or click the
variables shown below the field. Use the suggested variables instead of typing field names manually
when possible, so the prompt and greeting stay aligned with the selected template.

For outbound calls, these values come from the contact loaded into the campaign. For inbound calls,
these values are rendered only when the agent settings allow contact lookup from `Contact Inbound`
and `Contact Inbound` contains a loaded contact list with a contact matching the caller number.

Variables are replaced only when the key exists on the contact being used for that call. If a value
does not exist on the outbound contact, or on the inbound contact matched by `Contact Inbound`, that
variable is not rendered.

Before using custom template variables in outbound calls, make sure the campaign contacts include
the expected key-value pairs inside the `data` object.

See [Import Contacts](/run-workflows/import-contacts#associate-contacts-with-a-campaign-or-agent)
for how to upload a CSV and associate contacts with a campaign or an agent.

### Contact fields

Use contact fields when the action depends on imported or known customer data, such as:

- phone
- name
- company
- account reference

### Workflow variables

Use workflow variables when the action depends on configuration tied to the route, campaign, or
business process.

Examples:

- campaign name
- route identifier
- assigned queue
- target business unit

### Runtime values

Use runtime values when the agent learns something during the call that should shape the next
action.

Examples:

- interest level
- requested appointment time
- escalation reason
- follow-up outcome

## Real workflow examples

### Lead qualification

Pass:

- contact phone
- caller name
- qualification result
- requested follow-up type

### Appointment booking

Pass:

- service type
- preferred date
- preferred time window
- caller phone

### Routing decision

Pass:

- detected intent
- queue or destination identifier
- route context

### CRM update

Pass:

- contact id or phone
- call outcome
- notes collected during the call

## Keep dynamic parameters practical

Dynamic values are useful when they make the workflow more accurate. They are not useful when they
make the function hard to read or hard to debug.

Use them when:

- the same action is reused across multiple workflows
- the action depends on contact or runtime context
- the workflow would otherwise need many nearly identical functions

Avoid overcomplicating the input model with values that the external action does not actually need.

## Best practices

- pass only the parameters required for the action
- use clear names that show the business meaning of each input
- document where each value comes from
- keep prompt, function schema, and workflow context aligned
- test one workflow path at a time when validating dynamic mappings
